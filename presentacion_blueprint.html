El servidor Linux central de TecnoServicios S.L. sufre ataques de fuerza bruta incesantes hacia el usuario root en su acceso remoto (SSH). Depende exclusivamente de un sistema de usuario y contraseña tradicional.

SOLUCIÓN PLANTEADA

Implementar una capa extra de seguridad para proteger los datos de la empresa, pero con una condición innegociable:

NO SE PUEDE BLOQUEAR EL FLUJO DE TRABAJO OPERATIVO DEL EQUIPO TÉCNICO.

Soluciones Evaluadas

Claves SSH (Criptografía Asimétrica)

Ventajas:

Alta seguridad en el canal remoto. Elimina la contraseña del tránsito de red.

Desventajas:

No protege el login local si la clave privada es robada. Complejidad alta de gestión de claves para un equipo dinámico.

Descartada:demasiada fricción operativa para un equipo pequeño con rotación de personal.

Port Knocking

Ventajas:

Oculta el servicio SSH de los escáneres masivos.

Desventajas:

Seguridad por ocultación. Requiere crear y gestionar reglas dinámicas complejas en el firewall de Linux (UFW/iptables), lo cual dificulta la administración. Usabilidad muy baja.

Descartada:la gestión del firewall se volvía inmanejable y bloqueaba al equipo técnico.

2FA (TOTP) + Fail2Ban + Cambio de Puerto

Ventajas:

Seguridad real basada en dos factores. Protege login local y remoto a través de PAM. Bloqueo automático de bots. Flujo de trabajo rápido.

Desventajas:

Requiere que cada técnico lleve su app TOTP en el móvil. Necesita sincronización horaria (NTP).

Elegida: equilibrio óptimo entre seguridad, usabilidad y coste cero en licencias.
La Solución Adoptada: Tres Capas de Defensa
🚪
Capa 1 · Evasión
Puerto 52222

Trasladamos el servicio SSH fuera del puerto estándar TCP/22. Esto evade los rastreos masivos de botnets y herramientas como Shodan o Masscan.

🔒
Capa 2 · Prevención Activa
Fail2Ban + UFW

Un centinela que vigila los logs en tiempo real. Si detecta N intentos fallidos, modifica el cortafuegos (UFW) para bloquear la IP atacante durante 24 horas.

📱
Capa 3 · Autenticación Fuerte
MFA (PAM + TOTP)

La barrera final. Aunque un atacante conozca la contraseña, necesita un código temporal de 6 dígitos generado en el móvil del técnico.

Tecnología Clave: MODULOS PAM
🧩 ¿Qué son los modulos PAM?

Los módulos PAM son un marco flexible que centraliza la autenticación en Linux, actuando como una capa intermedia entre las aplicaciones (como SSH o sudo) y los mecanismos de seguridad. Permiten modificar las políticas de acceso (como añadir 2FA o restricciones) sin tener que recompilar ningún programa del sistema. Se configuran en /etc/pam.d/ mediante cuatro tareas clave: autenticación, gestión de cuentas, contraseñas y sesiones. Al ser componentes "intercambiables", permiten a los administradores añadir, quitar o combinar reglas de seguridad sobre la marcha de forma ágil y totalmente segura.

⚙️ ¿Cómo funciona?

Cuando un programa requiere verificar la identidad de un usuario, invoca a la biblioteca de PAM en lugar de comprobar las credenciales por sí mismo. PAM consulta el archivo de configuración en /etc/pam.d/ correspondiente a esa aplicación para saber exactamente qué reglas debe aplicar. Luego, ejecuta secuencialmente los módulos indicados para validar la contraseña, comprobar restricciones de cuenta o configurar la sesión. Finalmente, PAM recopila los resultados de todos los módulos y devuelve un veredicto de "éxito" o "fallo", permitiendo o denegando el acceso de forma transparente.

🔑 ¿Por qué es clave para nuestro proyecto?

Nos permite inyectar un paso extra de verificación (el código TOTP) en el proceso de login SSH sin modificar el propio demonio SSH. Añadimos el módulo pam_google_authenticator.so a la pila y PAM se encarga del resto.

Tecnología Clave: Fail2Ban
🛡️ ¿Qué es Fail2Ban?

Es un servicio de prevención de intrusiones (IPS) de código abierto que actúa como un centinela perimetral. Monitoriza continuamente los archivos de log del sistema en busca de ataques de fuerza bruta hacia el usuario root u otros vectores.

⚙️ ¿Cómo funciona?

Opera mediante «jaulas» (jails). Al detectar demasiados intentos fallidos desde una IP en un intervalo de tiempo, inyecta automáticamente una regla de bloqueo en el cortafuegos, mitigando la amenaza instantáneamente.

# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = 52222
maxretry = 3
logpath = %(sshd_log)s
backend = systemd
findtime = 10m
bantime = 24h
ignoreip = 127.0.0.1/8 ::1
✅ ¿Por qué lo hemos elegido?

✔️ Bloqueo automatizado: Veta IPs hostiles tras 3 accesos fallidos en el puerto 52222, repeliendo botnets en tiempo real.

✔️ Integración con UFW: Aplica de forma directa reglas de baneo temporal (24h) en el cortafuegos nativo del servidor.

✔️ Consumo eficiente: Solución open-source ultra-ligera (<15MB RAM) integrada en systemd sin degradar el rendimiento.

Flujo detallado de Fail2Ban

Interacción de Tecnologías: El Flujo de Acceso

Cuando un usuario (legítimo o atacante) intenta acceder al servidor, su conexión atraviesa secuencialmente las siguientes etapas. Si falla en cualquiera de ellas, el acceso es denegado de inmediato.

🚪
1. Petición TCP

El usuario abre conexión SSH apuntando al puerto 52222 (no al 22).

→
🛑
2. Filtro UFW + Fail2Ban

¿Está la IP en la lista de baneados? Sí → DROP. No → Continúa.

→
🔑
3. Autenticación Pública

¿La clave RSA es válida? No → Rechazo. Sí → Continúa.

→
📱
4. Desafío PAM (MFA)

sshd delega a PAM. Se solicita código TOTP al usuario.

🔢
5. Verificación TOTP

El módulo pam_google_authenticator.so valida el código contra ~/.google_authenticator.

→
✅
6. Acceso Concedido

Shell habilitada. El evento se registra en auth.log para auditoría.
